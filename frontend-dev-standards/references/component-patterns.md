# Component Patterns

用于新增或修改页面、组件、表单、列表、弹窗和交互。

## 组件设计

- 优先复用项目已有 UI primitives、业务组件、hooks 和类型。
- 组件 props 应表达业务意图，不暴露过多内部实现细节。
- 派生数据优先用局部变量或 memoized selector，避免重复 state。
- 副作用放在合适的 hook 或数据层，不混进纯展示组件。
- 复杂组件拆分以可读性为准，不为了形式拆成过多小文件。

## 用户状态

用户可见功能至少考虑：

- loading：首次加载、提交中、局部刷新。
- empty：无数据、筛选后无结果、无权限数据。
- error：请求失败、提交失败、部分失败。
- disabled：权限不足、表单无效、重复提交。
- long content：长标题、长用户名、多语言文本。
- navigation：刷新、返回、深链接、路由参数缺失。

## 表单

- 沿用项目已有表单库和校验方式。
- 提交时防重复提交，并给出明确反馈。
- 错误信息要靠近触发位置；全局错误可以用 toast 或 alert，但不要只写 console。
- 编辑场景要处理初始值、重置、脏数据、取消和保存失败。

## 列表与表格

- 保留筛选、排序、分页和搜索状态，除非产品明确要求重置。
- 明确空态和加载态，不让布局跳动。
- 批量操作要处理部分成功、权限限制和二次确认。
- 大数据列表优先查项目是否已有分页、虚拟滚动或无限加载模式。
