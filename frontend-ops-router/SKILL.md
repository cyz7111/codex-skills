# 前端运维总入口

## Overview

使用这个 skill 将前端任务先分流、定级和控风险，再进入具体执行 skill。它不负责直接写业务代码，而是帮助 Codex 在多项目、多技术栈、多平台的团队上下文中选择安全流程。

## 核心原则

- 先分类，再执行：不要在任务类型、项目范围和风险等级不清时直接改代码。
- 优先保护线上：只要用户描述线上异常、生产问题、紧急修复、历史遗留 bug、影响真实用户，就转入 `legacy-production-hotfix`。
- 项目事实优先：根据仓库文件、脚本、路由、接口、配置和用户描述判断，不输出空泛管理建议。
- 小步收敛：缺少项目信息时先做最小必要探测，再决定下一步。
- 高风险先提醒：涉及支付、登录、权限、环境配置、接口封装、公共组件、全局状态、构建发布时，先输出风险和影响面。

## 任务分流

按下面顺序判断任务类型：

1. 线上问题修复：
   - 用户提到线上、生产、紧急、故障、事故、历史遗留问题、用户反馈、某平台真实环境异常。
   - 进入 `legacy-production-hotfix`。
2. 发版或上线检查：
   - 用户提到发版、上线、灰度、回滚、go/no-go、检查清单、误发、漏发、回归。
   - 进入 `release-checklist-skill`。
3. 日常前端开发：
   - 用户要求新增或修改页面、组件、接口、状态、样式、交互、测试。
   - 进入 `frontend-dev-standards`。
4. 接手陌生项目或模块：
   - 用户要求先熟悉项目、梳理环境、接口、页面、配置，不急着改代码。
   - 读取 `references/task-routing.md` 的接手项目部分，输出项目考古计划。
5. 技术方案评审或任务拆解：
   - 用户要求评审方案、拆任务、排优先级、评估协作边界。
   - 读取 `references/task-routing.md` 的方案评审部分，输出风险和拆解建议。
6. 工程化或平台建设：
   - 用户提到 Monorepo、微前端、构建、CI/CD、公共包、规范、工具链。
   - 读取 `references/task-routing.md` 的工程化部分，先做现状调查和分阶段方案。

如果任务同时命中多个类型，按这个优先级处理：

```text
线上问题修复 > 发版检查 > 日常开发 > 接手项目 > 技术方案 > 工程化建设
```

## 最小探测

当仓库上下文不足以分流时，先做最小探测：

- 读取 `package.json`、项目 README、构建脚本、路由入口、页面目录、接口目录、配置目录。
- 使用 `rg --files` 和 `rg` 查找关键字，不全量阅读无关文件。
- 识别技术栈、运行平台、业务线、影响页面、接口和配置。
- 如果是多仓库或缺少当前项目文件，只基于用户描述输出待确认项。

## 风险定级

将任务粗分为：

- P0：线上故障、支付/登录/核心链路异常、数据错误、生产不可用。
- P1：交付阻断、发版阻断、主要业务路径异常。
- P2：正常需求、明确范围内的页面或接口调整。
- P3：工程化优化、规范建设、体验优化、非紧急治理。

P0/P1 任务必须先输出影响面、验证路径和回滚考虑，再修改代码。

## 输出格式

分流完成后，优先用这个结构回复：

```markdown
**任务判断**
类型：
优先级：
涉及项目/平台：

**建议工作流**
应使用的 skill：
原因：

**已知风险**
- 

**下一步**
-
```

当用户明确要求执行任务时，不要停在分流结果；继续进入对应 skill 的流程。
