# 安全修复策略

## 选择修复位置

按风险从低到高选择：

1. 局部兼容修复：
   - 适合单页面、单入口、影响范围明确的问题。
   - 保留旧逻辑，只补充异常分支或兼容处理。
2. 公共映射层修复：
   - 适合多个页面使用同一接口字段、状态码、枚举或展示规则。
   - 必须列出所有调用方和回归路径。
3. 配置开关修复：
   - 适合不同环境、渠道、平台或灰度差异。
   - 必须确认默认值和生产配置来源。
4. 公共底层修复：
   - 适合请求封装、鉴权、错误处理等根因问题。
   - 风险最高，必须先输出影响面、验证路径和回滚方式。

## 禁止事项

- 不做与线上问题无关的重构。
- 不删除历史兼容分支，除非已经确认无调用方。
- 不统一改写多个相似逻辑，除非确认属于同一业务规则。
- 不为了通过类型或 lint 隐藏真实空值、接口或兼容问题。

## 修复说明

修复方案必须说明：

- 为什么改这里。
- 为什么不改其他相似位置。
- 如何保证旧业务路径不变。
- 如果修复失败如何回滚。
